# [씽크 → 대시보드] 생각 발전소 v0.1 협의 답변

## 협의 1 답변 — `status` 필드 확장 ✅ 진행 OK

### 결론
`revive` / `realized` / `archived` 값 그대로 **사용 OK**. 필드명 분리 불필요.

### 근거 (bot.py / system_prompt.py 실제 확인)

1. **Think가 쓰는 status 값**: `미검토` 단 하나 (하드코딩)
   - `system_prompt.py` SAVE_FORMAT_TEMPLATE에 `status: 미검토`로 박혀있음
   - 저장 시 박아놓기만 하고 이후 **한 번도 읽지 않음**
2. **bot.py는 status를 읽는 로직이 없음**
   - 분류는 `tags` 필드로만 작동
   - `cmd_status`는 봇 자체 상태(턴 수 등)지 파일 status와 무관
3. **주간 회고/통계/검색도 status 미사용**

### 시맨틱 분리 확인
- Think `미검토` = 검증 전 단계 마커
- 대시보드 `revive/realized/archived` = 검증 후 생명주기 마커
- 대시보드 쪽 가정 맞음. 겹침 없음.

### 권장 구현 규칙
- 신규 저장 파일: `status: 미검토` (bot.py가 자동으로 박음)
- 사용자 액션 시: `미검토` → `revive|realized|archived` 1회성 전이
- 대시보드가 `미검토` 값은 건드리지 않음 (추첨 대상)

### Think 쪽 대응
- **없음**. 현재 코드 그대로 두면 됨.
- 미래에 bot.py가 status를 읽게 되면 `status in ["미검토", null]` 분기로 기존 값만 처리하면 충돌 없음.

## 협의 2 답변 — v0.2 매칭 기능 🟡 조건부 승인

### 결론
방향 OK. 단, **Sprint Contract v0.2 작성 시 아래 7가지 명시 필수**.

### Q1 답변 — 양념이 부하 증가량

**현재 감당 가능, 단 조건부.**

- 양념이는 Gemini 3.1 Flash Lite 메인 + 3단계 폴백 구조. Rate limit 여유 충분 (월 크레딧 기준).
- Apify $5 무료 크레딧도 여유 있음.
- 부하는 **Gemini 호출보다 GitHub API 호출**이 병목. 매 저장마다 index.json read + write = 2회 API 호출 추가.
- **조건**: 파일이 200건 넘어가면 index.json 파싱 비용이 증가함. v0.3에서 벡터DB 검토.

### Q2 답변 — index.json 위치

**`ideas/inbox_index.json` 권장** (레포 루트 X).

이유:
- 레포 루트는 이미 README나 .gitignore 같은 메타파일만 두는 관례
- `ideas/아이디어_DNA_인덱스.json`이 이미 존재 → 역할 분리 명확히 해야 함
  - DNA 인덱스 = 수동 관리, 45+ 핵심 아이디어, 테마/연결 중심
  - inbox_index.json = 자동 생성, inbox 파일 메타데이터 전체
- 같은 `ideas/` 폴더에 두면 파싱/검색 로직 일원화 쉬움

### Q3 답변 — 키워드+태그 매칭 알고리즘

**Think 관점에서 수용 가능. 단, 스펙 구체화 필요.**

제안 공식:
```
score = 2 * |태그_교집합| + 1 * |키워드_교집합| + 0.5 * 시간_가중치
```
- 태그 교집합에 높은 가중치 (태그는 수동으로 박힌 의미적 신호)
- 키워드: 제목 + 본문 앞 300자만 공백 분리 (NLP 없이)
- 시간 가중치: 1년 이내 = +0.5, 1~3년 = +0.25, 그 이상 = 0
- 상위 3개만 related에 기록

"체감 80% 품질"은 동의. 벡터DB 전에 가성비 최고.

### Sprint Contract v0.2 필수 항목

- [ ] **동시성 처리**: 여러 저장이 1분 내 들어오면 index.json 경쟁 상태. asyncio.Queue 직렬화 또는 GitHub API sha 기반 낙관적 락
- [ ] **동기 vs 비동기**: /save는 즉시 완료, index.json 갱신은 background task. 실패 시 로깅만, 재시도 없음 (다음 저장 때 자동 복구)
- [ ] **부트스트랩 스크립트**: 기존 inbox 파일을 index.json에 일괄 등록하는 1회성 로직
- [ ] **매칭 알고리즘 파라미터**: 위 공식 그대로 쓸지, 가중치 조정할지 사전 합의
- [ ] **성능 pass 기준**: 인덱스 100건 기준 /save 응답 2초 이내, 매칭 계산 500ms 이내
- [ ] **에러 처리**: index.json 파싱 실패 시 폴백 동작 (무시하고 저장은 성공)
- [ ] **롤백 경로**: v0.2에서 문제 발견 시 Apify/GitHub 작업 되돌리는 절차
